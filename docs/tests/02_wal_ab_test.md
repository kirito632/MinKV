# WAL A/B 测试

> 对比三种模式的性能：纯内存 vs Group Commit vs 同步刷盘。

---

## 环境准备

参考 [00_environment_setup.md](./00_environment_setup.md)。

---

## 编译

```bash
cd ~/MinKV
bash build_wal_ab_test.sh
```

看到 `✓ 编译成功！` 即可。

```bash
cd ~/MinKV
./build/bin/wal_ab_test
```

总耗时约 3~5 分钟，结果同时保存到 `wal_ab_test_results.csv`。

## 测试场景参数

测试代码：`MinKV/src/tests/wal_ab_test.cpp`

| 参数 | 值 |
|------|-----|
| 分片数 | 32 |
| 每分片容量 | 10,000 |
| 预填充数据量 | 100,000 条 |
| Key 范围 | 0 ~ 999,999（100万） |
| 每线程操作数 | 50,000 次（同步刷盘模式只测 1,000 次） |
| 读写比 | 90% GET + 10% PUT |
| 线程数 | 1 / 2 / 4 / 8 |

### 三种测试模式

| 模式 | fsync_interval | 说明 |
|------|---------------|------|
| 纯内存 | 不开 WAL | 性能基准，无持久化 |
| Group Commit | 10ms | 每 10ms 批量 fsync |
| 同步刷盘 | 0ms | 每次写操作立即 fsync，最慢 |

> 同步刷盘模式因为极慢，只在 1/2 线程时测试，且每线程只跑 1,000 次。

---

## 测试结果

### 各线程数对比

| 线程 | 纯内存 QPS | Group Commit QPS | 损耗 | 同步刷盘 QPS |
|------|-----------|-----------------|------|------------|
| 1 | 278万 | 224万 | 19.5% | 1.6万 |
| 2 | 383万 | 306万 | 20.3% | 1.6万 |
| 4 | 378万 | 323万 | 14.5% | - |
| 8 | 315万 | 275万 | 12.8% | - |

### 关键结论

- 纯内存峰值：**383万 QPS**（2线程）
- Group Commit 损耗控制在 **13~20%**
- Group Commit 相比同步刷盘提升约 **198 倍**

---

## 背景说明

早期实现用 `iostream::flush()` 只刷到 OS page cache，测出的损耗是假的 8.7%。
后来改为 POSIX `write()` + `fsync()`，数据真正落盘，损耗才是真实的 13~20%。
这个发现本身说明了性能测试要验证实现的正确性，不能只看数字。
