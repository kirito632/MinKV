# 综合性能测试 (Comprehensive Benchmark)

> 测试意图：验证系统在不同并发度和读写比下的吞吐量与延迟表现，找到最优工作点，同时测量向量检索的实际性能。具体验证三个问题：
> 1. 多线程扩展性如何？几个线程时达到峰值？
> 2. 读写比对性能的影响有多大？
> 3. 向量检索在真实系统中的延迟是多少？

## 环境准备

### 服务器信息
- 腾讯云竞价实例，每次重启 IP 变化、环境重置
- 用户名: `ubuntu`，当前 IP: `172.81.247.190`（重启后需更新）

### 首次环境搭建
```bash

sudo apt update
sudo apt install -y cmake g++ make

# 验证版本
cmake --version   # 需要 >= 3.10
g++ --version     # 需要支持 C++17
```

### 上传代码
```bash
# 本地执行，上传整个 MinKV 目录
scp -r MinKV/ ubuntu@<服务器IP>:~/MinKV/
```

### 编译
```bash
# 服务器上执行
cd ~/MinKV
mkdir -p build && cd build
cmake -DCMAKE_BUILD_TYPE=Release \
      -DCMAKE_CXX_FLAGS="-O2 -march=native -mavx2 -mfma" ..
make -j$(nproc) comprehensive_benchmark
```

## 测试场景

### 测试代码
`MinKV/src/tests/comprehensive_benchmark.cpp`

### 参数配置
| 参数 | 值 |
|------|-----|
| 分片数 | 32 |
| 每分片容量 | 10,000 |
| 预填充数据量 | 100,000 条 |
| Key 范围 | 0 ~ 999,999（100万） |
| 每线程操作数 | 100,000 次 |
| 延迟采样间隔 | 每 100 次操作采样 1 次 |

### 子测试 1：多线程并发性能（R90W10）
- 线程数：1 / 2 / 4 / 8 / 16
- 读写比：90% GET + 10% PUT

### 子测试 2：不同读写比例（固定 8 线程）
- 读写比：R50W50 / R70W30 / R90W10 / R95W5 / R99W1

### 子测试 3：向量检索性能
- 预填充 100,000 条 128 维向量
- 每线程执行 500 次 Top-10 向量检索
- 线程数：1 / 2 / 4 / 8

## 关键结论

1. 峰值 5.08M QPS（2线程），P99 延迟 2.81μs
2. 2线程后性能下降，原因是此测试为内存密集型（key范围100万，工作集超出L3 cache），线程越多 cache 争用越激烈，cache miss 增加导致吞吐下降。计算密集型任务才是"核数越多越好"，KV存储的瓶颈在内存带宽而非CPU算力
3. 读比例越高性能越好，R99W1 比 R50W50 高约 13%
4. 向量检索受限于暴力搜索 O(N)，10万向量下单次检索约 12ms
