# 综合性能测试 (Comprehensive Benchmark)

> 测试意图：验证系统在不同并发度、读写比和命中率下的吞吐量与延迟表现，同时测量 Noop baseline（框架开销）和向量检索性能。
>
> 测试命令：`taskset -c 0,2,4,6 ./bin/comprehensive_benchmark`

---

## 环境准备

### 首次环境搭建
```bash
# 安装依赖
sudo apt update
sudo apt install -y cmake g++ make

# 验证版本
cmake --version   # 需要 >= 3.10
g++ --version     # 需要支持 C++17
```

### 上传代码
```bash
# 本地执行，上传整个 MinKV 目录
scp -r MinKV/ ubuntu@<服务器IP>:~/MinKV/
```

---

## 编译

> **注意**：`-mavx2 -mfma` 已在 [`CMakeLists.txt`](../../CMakeLists.txt:13) 中通过 `add_compile_options` 全局添加，编译时无需重复传入。

### O2 编译（默认）
```bash
# 服务器上执行
cd ~/MinKV
mkdir -p build && cd build
cmake -DCMAKE_BUILD_TYPE=Release \
      -DCMAKE_CXX_FLAGS="-O2 -march=native -mavx2 -mfma" ..
make -j$(nproc) comprehensive_benchmark
```

### O3 编译（方案一：推荐，让 CMake Release 模式自动使用 -O3）
```bash
cd ~/MinKV/build
cmake -DCMAKE_BUILD_TYPE=Release ..
make -j$(nproc) comprehensive_benchmark
```

### O3 编译（方案二：显式指定，效果等价）
```bash
cd ~/MinKV/build
cmake -DCMAKE_BUILD_TYPE=Release \
      -DCMAKE_CXX_FLAGS="-O3 -march=native -mavx2 -mfma" ..
make -j$(nproc) comprehensive_benchmark
```

> **说明**：`CMAKE_BUILD_TYPE=Release` 对 GCC/Clang 默认添加 `-O3 -DNDEBUG`。如果同时手动传入 `CMAKE_CXX_FLAGS`，会**覆盖** CMake 的默认 Release 标志，因此 O2 测试需要显式传 `-O2`，而 O3 测试建议不传 `CMAKE_CXX_FLAGS`，让 CMake 自动处理。

cd ~/MinKV/build

## 测试场景与参数

### 参数配置

| 参数 | 值 |
|------|-----|
| 分片数（默认） | 32 |
| 每分片容量 | 10,000 |
| 预填充数据量 | 100,000 条 |
| Key 范围 | 0 ~ 999,999（100万） |
| 每线程操作数 | 100,000 次 |
| 延迟采样间隔 | 每 100 次操作采样 1 次 |

### 实验 A：Noop Baseline
- 用空函数替换 `cache.get()`，测量 benchmark 框架自身扩展性
- 线程数：1 / 2 / 4 / 8 / 16

### 实验 B-C：多线程并发性能（R90W10）
- 线程数：1 / 2 / 4 / 8 / 16
- 读写比：90% GET + 10% PUT
- 两种命中率场景：
  - **Hit-heavy**：`preload_count == key_range` ≈ 100% 命中
  - **Miss-heavy**：`preload_count << key_range` ≈ ~10% 命中

### 实验 D：不同读写比例（固定 8 线程，miss-heavy）
- 读写比：R50W50 / R70W30 / R90W10 / R95W5 / R99W1

### 实验 E：向量检索性能
- 预填充 100,000 条 128 维向量
- 每线程执行 500 次 Top-10 向量检索
- 线程数：1 / 2 / 4 / 8

### 实验 F：分片数扩展性测试
- 固定 8 线程，hit-heavy，R90W10
- 分片数：1 / 4 / 16 / 32 / 64 / 128 / 256

### 实验 A：Noop Baseline

Noop_Baseline   1      21,519,079     0.03        0.03        0.03      0%     0
Noop_Baseline   2      43,516,944     0.03        0.03        0.03      0%     0
Noop_Baseline   4      85,605,115     0.03        0.03        0.03      0%     0

1→2→4 接近线性扩展（2.02x / 3.98x），8→16 受限于物理核数（4核 SMT）但继续增长，benchmark 框架无扩展性问题。

### 实验 B-C：多线程并发 R90W10（32 分片）

**Hit-heavy（~99% 命中）：**

| 线程数 | QPS | P50(us) | P95(us) | P99(us) | 命中率 |
|--------|-----|---------|---------|---------|--------|
| 1 | 4,490,822 | 0.17 | 0.39 | 0.55 | 99% |
| 2 | 7,548,174 | 0.19 | 0.37 | 1.09 | 99% |
| 4 | 12,919,125 | 0.19 | 0.45 | 3.62 | 99% |
| 8 | 15,340,790 | 0.18 | 0.45 | 8.39 | 99% |
| 16 | 15,344,934 | 0.18 | 0.46 | 6.76 | 99% |

**Miss-heavy（~10% 命中）：**

| 线程数 | QPS | P50(us) | P95(us) | P99(us) | 命中率 |
|--------|-----|---------|---------|---------|--------|
| 1 | 5,705,375 | 0.14 | 0.29 | 0.47 | 10% |
| 2 | 7,990,507 | 0.16 | 0.40 | 1.94 | 10% |
| 4 | 10,874,417 | 0.19 | 0.54 | 6.47 | 11% |
| 8 | 10,004,602 | 0.21 | 1.29 | 14.25 | 13% |
| 16 | 10,140,707 | 0.22 | 1.49 | 29.39 | 16% |

1→4 线程扩展比：hit-heavy 2.88x / miss-heavy 1.91x，说明全局锁（`health_mutex_` / `global_consistency_lock_`）已被移除，shard 并行性有效释放。

### 实验 D：不同读写比例（8 线程，miss-heavy）

| 场景 | QPS | P50(us) | P95(us) | P99(us) | 命中率 |
|------|-----|---------|---------|---------|--------|
| R50W50 | 5,840,292 | 0.44 | 2.89 | 23.31 | 24% |
| R70W30 | 7,650,069 | 0.28 | 1.69 | 18.31 | 20% |
| R90W10 | 9,677,814 | 0.21 | 1.41 | 16.36 | 13% |
| R95W5 | 13,795,786 | 0.19 | 0.53 | 9.79 | 11% |
| R99W1 | 15,117,668 | 0.17 | 0.45 | 7.96 | 10% |

写占比从 50% 降到 1%，QPS 提升 **2.59x**（5.8M→15.1M），这是正常可扩展缓存系统的典型行为。写锁（per-shard `std::mutex`）开销主导了低频读写比下的性能差异。

### 实验 E：向量检索（10 万向量，128 维，Top-10）

| 线程数 | QPS | P50(us) | P95(us) | P99(us) |
|--------|-----|---------|---------|---------|
| 1 | 68 | 13,988.76 | 19,561.78 | 22,835.47 |
| 2 | 80 | 24,745.22 | 33,383.99 | 38,726.83 |
| 4 | 82 | 47,005.63 | 75,909.65 | 93,750.86 |
| 8 | 83 | 92,087.60 | 161,900.86 | 195,683.76 |

向量检索在多线程下几乎不扩展（68→83 QPS），这是独立的 **memory bandwidth bound** 问题——全表扫描 100K × 128 floats 已触达内存带宽上限，与锁竞争无关。

### 实验 F：分片数扩展性（8 线程，hit-heavy）

| 分片数 | QPS | P50(us) | P95(us) | P99(us) | 命中率 |
|--------|-----|---------|---------|---------|--------|
| 1 | 3,734,338 | 0.52 | 9.99 | 20.25 | 10% |
| 4 | 8,239,203 | 0.24 | 4.18 | 14.02 | 40% |
| 16 | 12,792,671 | 0.19 | 0.61 | 12.79 | 99% |
| 32 | 15,772,756 | 0.18 | 0.45 | 5.58 | 99% |
| 64 | 16,872,105 | 0.18 | 0.39 | 1.56 | 99% |
| 128 | 18,181,753 | 0.18 | 0.30 | 0.46 | 99% |
| 256 | 18,235,950 | 0.18 | 0.29 | 0.44 | 99% |

**Shard Scaling Pattern（三阶段）：**

| 阶段 | 分片范围 | 特征 |
|------|---------|------|
| Phase 1 — Lock Bound | 1→16 | 陡峭上升：从全局互斥锁中解放 |
| Phase 2 — Core Bound | 16→128 | 缓和上升：接近 4 物理核吞吐上限 |
| Phase 3 — Cache/Allocator Bound | 128→256 | 平台期：瓶颈转移至 cache miss / allocator |

P99 从 20.25μs（1 shard）降至 0.44μs（256 shards），**46 倍改善**，锁竞争被充分消除。

## O2 vs O3 对比分析

| 实验场景 | O2 | O3 | 差异 |
|----------|-----|-----|------|
| Noop 8T | 58.6M | 86.6M | O3 +48%（框架级，计算密集路径受益） |
| Hit-heavy 8T | 15.3M | 15.4M | ≈ 相同 |
| Miss-heavy 8T | 10.0M | 10.4M | O3 +4% |
| R99W1 8T | 15.1M | 15.5M | ≈ 相同 |
| Vector 1T | 68 QPS | 70 QPS | ≈ 相同 |

**结论**：O2 与 O3 在真实 workload 下的差距 <5%，瓶颈不在编译器优化级别，而在同步开销、哈希查找和内存访问延迟。O3 的优势仅在对计算密集的纯框架路径（Noop）中体现。

---

## 关键结论

1. **Benchmark 框架无扩展性问题**：Noop baseline 1→4→8 接近线性扩展。
2. **32 分片下 hit-heavy 8 线程达 ~15.3M QPS（O2）**，多线程扩展性恢复良好（1→4 线程 2.88x）。
3. **读写比扩展正常**：R99W1 比 R50W50 高 2.59x，写锁开销是性能差异的主要来源。
4. **向量检索是独立的 memory bandwidth bound 问题**，多线程下几乎不扩展（68→83 QPS）。
5. **分片数扩展呈三阶段模式**：1→16 Lock Bound（陡峭）、16→128 Core Bound（缓和）、128→256 平台期（瓶颈迁移至 cache/allocator）。
6. **P99 从 1 shard ~20μs 降至 256 shards ~0.44μs**（46 倍改善），锁竞争被充分消除。
7. **O2 vs O3 差距 <5%**，瓶颈不在编译优化，而在同步和内存访问。
